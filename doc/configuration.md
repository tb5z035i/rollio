# Rollio Configuration Reference

This document describes every field in the Rollio configuration file
(`rollio_config.yaml`).  The file is generated by the setup wizard
(`rollio setup`) and consumed by the data-collection TUI (`rollio collect`).

You can also hand-edit the YAML or use the template at
[`doc/rollio_config.template.yaml`](rollio_config.template.yaml) as a
starting point.

---

## How the config is created

```
rollio setup [-o OUTPUT] [-f] [--sim-cameras N] [--sim-arms N]
```

| Flag | Default | Description |
|------|---------|-------------|
| `-o`, `--output` | `rollio_config.yaml` | Path to write the config file |
| `-f`, `--force` | off | Overwrite an existing config |
| `--sim-cameras` | `0` | Number of simulated cameras to include during setup |
| `--sim-arms` | `0` | Number of simulated robot arms to include during setup |

By default, setup only shows detected real hardware.  For local testing,
add `--sim-cameras` and/or `--sim-arms` to make simulated devices appear.

The wizard walks through five screens:

1. **Cameras** — scans for V4L2 and RealSense cameras, plus any requested
   simulated cameras. For each
   detected device, a live ASCII preview is shown.  You assign a channel
   name, choose pixel format and resolution.
2. **Robots** — scans for AIRBOT Play arms (via CAN bus), plus any
   requested simulated arms. For
   each detected device, joint positions are displayed live.  You assign a
   channel name and role (leader / follower).
3. **Project settings** — project name, storage root, collection mode,
   RGB codec, depth codec, and PlotJuggler streaming toggle.
4. **Tele-op pairing** — shown only for `mode: teleop`.  You explicitly
   choose which leader controls which follower, and whether the pair uses
   direct joint mapping or FK-IK pose mapping.
5. **Summary** — live preview of all selected cameras and robots.
   Press Enter to save, Esc to cancel.

After saving, start data collection with:

```
rollio collect --config rollio_config.yaml
```

## Top-level fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `project_name` | string | `"default"` | Name of the dataset project. Used as subdirectory under `storage.root`. |
| `description` | string | `""` | Free-text description of the dataset or task. |
| `fps` | int | `30` | Global target sampling rate (frames per second) for episode recording. All sensors are polled at this rate. |
| `mode` | string | `"teleop"` | Data-collection mode. `"teleop"` for tele-operation (leader/follower), `"intervention"` for human intervention. |
| `plotjuggler_enabled` | bool | `false` | When enabled, each robot joint-state read publishes best-effort UDP JSON telemetry to PlotJuggler on `localhost:${PLOTJUGGLER_PORT:-9870}`. |

### Example

```yaml
project_name: pick_and_place
description: "Picking red blocks from tray A to bin B"
fps: 30
mode: teleop
plotjuggler_enabled: false
```

## `cameras` — camera configuration

A list of camera entries.  Each entry represents one image stream that will
be recorded as a video track per episode.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | string | `"cam0"` | Channel name.  Used in file paths (`videos/chunk-000/{name}/episode_*`) and LeRobot feature keys (`observation.images.{name}`). Must be unique. |
| `type` | string | `"pseudo"` | Camera driver type.  One of `"pseudo"`, `"v4l2"`, `"realsense"`. |
| `device` | int or string | `0` | Device identifier.  For V4L2: device index (e.g. `0` → `/dev/video0`).  For RealSense: serial number only (e.g. `"335522070371"`). |
| `width` | int | `640` | Frame width in pixels. |
| `height` | int | `480` | Frame height in pixels. |
| `fps` | int | `30` | Camera capture framerate. Should be ≥ the global `fps`. |
| `pixel_format` | string | `"rgb24"` | Pixel format / fourcc code for the capture stream. Common values listed below. |
| `id_path` | string | `""` | udev `ID_PATH` string for stable device identification across reboots (V4L2 only). Populated automatically by the scanner. |
| `channel` | string | `"color"` | For RealSense cameras, which stream to use: `"color"`, `"depth"`, or `"infrared"`. Ignored for other camera types. |

### Camera types

#### `pseudo`

Synthetic camera that renders incrementing frame numbers.  Useful for
testing pipelines without real hardware.

- `device`: ignored (any value accepted)
- `pixel_format`: `"rgb24"`

#### `v4l2`

USB webcam captured via Video4Linux2.

- `device`: V4L2 device index (integer).  `0` corresponds to
  `/dev/video0`, `2` to `/dev/video2`, etc.
- `pixel_format`: `"MJPG"` (recommended for USB bandwidth) or `"YUYV"`.
  The setup wizard lists all formats reported by `v4l2-ctl`.
- `id_path`: udev `ID_PATH`, for example
  `"pci-0000:af:00.0-usb-0:4:1.0"`.  This allows re-identifying the
  same physical camera after a reboot even if the device index changes.

#### `realsense`

Intel RealSense camera (D400/L500 series).

Each physical RealSense unit exposes up to three independent streams
(color, depth, infrared).  In the config, each stream is a separate
camera entry with the same serial but a different `channel`.

- `device`: serial number only, for example `"335522070371"`.
- `channel`: `"color"`, `"depth"`, or `"infrared"`.
- `pixel_format`: depends on channel:
  - Color: `"bgr8"`, `"rgb8"`, `"yuyv"`
  - Depth: `"z16"`
  - Infrared: `"y8"`, `"y16"`

### Common pixel formats

| Format | Description | Typical use |
|--------|-------------|-------------|
| `rgb24` | 24-bit RGB | Pseudo camera |
| `MJPG` | Motion JPEG | V4L2 USB cameras (saves USB bandwidth) |
| `YUYV` | YUV 4:2:2 packed | V4L2 USB cameras (uncompressed) |
| `bgr8` | 24-bit BGR | RealSense color |
| `z16` | 16-bit unsigned depth (millimeters) | RealSense depth |
| `y8` | 8-bit grayscale | RealSense infrared |
| `y16` | 16-bit grayscale | RealSense infrared |

### Example

```yaml
cameras:
  - name: wrist_cam
    type: v4l2
    device: 0
    width: 640
    height: 480
    fps: 30
    pixel_format: MJPG
    id_path: pci-0000:af:00.0-usb-0:4:1.0
    channel: color

  - name: overhead_color
    type: realsense
    device: "335522070371"
    width: 1920
    height: 1080
    fps: 30
    pixel_format: bgr8
    id_path: ""
    channel: color

  - name: overhead_depth
    type: realsense
    device: "335522070371"
    width: 1280
    height: 720
    fps: 30
    pixel_format: z16
    id_path: ""
    channel: depth
```

## `robots` — robot arm configuration

A list of robot arm entries.  Each entry represents one proprioception
channel that will be recorded as columns in the episode Parquet files.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | string | `"arm0"` | Channel name.  Used in Parquet column names (`observation.state.{name}.position`, etc.). Must be unique. |
| `type` | string | `"pseudo"` | Robot driver type.  One of `"pseudo"`, `"airbot_play"`, `"nero"`. |
| `role` | string | `"follower"` | Arm role for tele-operation.  `"leader"` arms are gravity-compensated and manually guided; `"follower"` arms replicate the leader's motion. |
| `num_joints` | int | `6` | Number of joints (degrees of freedom). |
| `device` | string | `""` | Device identifier.  For AIRBOT Play: CAN interface name (e.g. `"can0"`). For pseudo: ignored. |

### Robot types

#### `pseudo`

Synthetic 6-DOF arm that generates sine-wave joint motion.  Useful for
testing without real hardware.

#### `airbot_play`

AIRBOT Play robotic arm, communicating over CAN bus.  Provides joint
position, velocity, and effort feedback at hardware rate.

Requires optional dependencies:

```
pip install rollio[airbot]
```

- `device`: CAN interface name, e.g. `"can0"`, `"can1"`.
- `num_joints`: typically `6`.

#### `nero`

AgileX NERO arm (planned).  Not yet implemented.

### Roles in tele-operation

In `mode: teleop`, arms are grouped into leader/follower pairs:

- **Leader** — gravity-compensated, operator drags it by hand.  Joint
  positions are recorded as the action signal.
- **Follower** — replicates the leader's joint positions in real time.
  Joint positions are recorded as the observation state.

Typical setups:

| Setup | Arms |
|-------|------|
| Single-arm | 1 leader + 1 follower |
| Bimanual | 2 leaders + 2 followers (left/right) |

### Example

```yaml
robots:
  - name: follower_arm
    type: airbot_play
    role: follower
    num_joints: 6
    device: can0

  - name: leader_arm
    type: airbot_play
    role: leader
    num_joints: 6
    device: can1
```

## `teleop_pairs` — explicit tele-operation relationships

When `mode: teleop`, setup writes an explicit list of pairings so you can
control which leader drives which follower and which mapping each pair uses.
Configs with both leaders and followers should define these pairs explicitly;
roles alone are no longer treated as an implicit pairing rule.

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Unique pair name |
| `leader` | string | Name of the robot configured as the leader |
| `follower` | string | Name of the robot configured as the follower |
| `mapper` | string | `"joint_direct"` or `"pose_fk_ik"` |

`joint_direct` is the default when leader/follower robots are compatible
same-type robots with matching joint counts.  Otherwise, setup defaults the
pair to `pose_fk_ik`.

### Example

```yaml
teleop_pairs:
  - name: pair_0
    leader: left_leader
    follower: left_follower
    mapper: joint_direct

  - name: pair_1
    leader: right_leader
    follower: right_follower
    mapper: pose_fk_ik
```

## `encoder` — RGB/depth export codecs

Controls the codecs used when episode videos are exported.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `video_codec` | string | `"libx264"` | Codec for RGB/color streams |
| `depth_codec` | string | `"ffv1"` | Codec for depth/grayscale streams |
| `background_workers` | int | `1` | Background export workers |

The setup wizard shows only codecs that are usable on the current machine.
For RGB streams, it prefers the best available hardware H.264 encoder first,
then software H.264, then MPEG-4. Depth/grayscale streams default to
lossless codecs.

### Example

```yaml
encoder:
  video_codec: h264_nvenc
  depth_codec: ffv1
  background_workers: 1
```

## `storage` — data output settings

Controls where recorded episodes are written.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `root` | string | `"~/rollio_data"` | Root directory for all datasets.  The actual data lives under `{root}/{project_name}/`. Tilde is expanded at runtime. |
| `lerobot_version` | string | `"v2.1"` | LeRobot dataset format version. Only `"v2.1"` is currently supported. |

### LeRobot v2.1 directory layout

After recording, the output directory looks like:

```
~/rollio_data/{project_name}/
├── meta/
│   └── info.json                         # Dataset schema and metadata
├── data/
│   └── chunk-000/
│       ├── episode_000000.parquet        # Tabular data for episode 0
│       ├── episode_000001.parquet
│       └── ...
└── videos/
    └── chunk-000/
        ├── {camera_name}/
        │   ├── episode_000000.mp4        # Video for episode 0
        │   ├── episode_000001.mp4
        │   └── ...
        └── {another_camera}/
            └── ...
```

### Example

```yaml
storage:
  root: ~/rollio_data
  lerobot_version: v2.1
```

## `controls` — keyboard bindings

Controls the keyboard shortcuts used during data collection.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `start_stop` | string | `" "` (space) | Key to start or stop recording an episode. |
| `keep` | string | `"k"` | Key to keep the most recently recorded episode (write to disk). |
| `discard` | string | `"d"` | Key to discard the most recently recorded episode. |

### Data-collection workflow

1. Press `start_stop` to begin recording an episode.
2. Perform the task (tele-operate or let the robot execute).
3. Press `start_stop` again to stop recording.
4. Review, then press `keep` to save or `discard` to throw away.
5. Repeat.

### Example

```yaml
controls:
  start_stop: " "
  keep: k
  discard: d
```

## Validation

The config is validated at load time using Pydantic.  Invalid values
(wrong types, unknown `mode`, etc.) will raise a clear error.  You can
validate a config without starting collection:

```python
from rollio.config.schema import RollioConfig

cfg = RollioConfig.load("rollio_config.yaml")
print(cfg.model_dump())
```

Or from the CLI, simply run `rollio collect --config <path>` — it will
report any errors before entering the TUI.
